Global design rules

Six rules applied to every JSON file. They make the apps naturally interoperable without rework later.

Rule 1 — IDs
Every entity gets a stable, readable ID. Recipes: rcp_001. Ingredients: ing_042. Trips: trp_2025_amsterdam. Race events: evt_paris_marathon_2026. Never bare integers — they survive copy/paste between files and mean something when read cold. Your CSV uses bare integers for Recipe_ID; pad to rcp_001 format during conversion.
Rule 2 — Dates are always ISO 8601
Full date: "2025-05-09"  ·  Year-month: "2025-05". Never "Jan", "May 2025", or "09/05/25". Your calorie schedule CSV already uses ISO dates — good. Your budget CSV uses separate Year / abbreviated Month columns — merge to "2017-01" during conversion.
Rule 3 — Numbers are numbers
Calories, grams, costs are always numeric, never strings. 520 not "520". Missing/unknown = null. Crucially: 0 is a real value (your recipes have real zero Trans Fat, Cholesterol etc.) — never replace a zero with null.
Rule 4 — Every file has a meta block
The homepage reads only meta — a small pre-computed summary at the top of each file. It never scans the full dataset. Meta is updated whenever data changes and holds pre-calculated values: today's kcal target, current month variance, days to next trip.
Rule 5 — Cross-references use IDs, not copies
A fuel schedule day references a recipe as "recipeId": "rcp_012". A trip references a race as "eventId": "evt_paris_marathon_2026". One source of truth, referenced everywhere. Changing a recipe name in recipes.json propagates everywhere automatically.
Rule 6 — Tags are the integration glue
Tags are lowercase-hyphenated string arrays. The same tag "paris-2026" on a trip, a budget month, and a training event is what lets the homepage connect them into a single view. Establish your tag vocabulary early and use it consistently.

Kitchen

The Kitchen

Two CSV files become three JSON files: a master ingredient library, then recipes that reference it. The ingredient library is the key design decision — it eliminates duplication and powers future features like shopping lists and "which recipes use X?"

Observed from CSV — recipes-source
20 columns: Recipe_ID (int), Source, Cuisine, Sub-Category, Dish Name, Servings, then 13 nutrition columns (kcal, full fat breakdown: total/sat/mono/poly/trans, cholesterol, sodium, potassium, carbs, fibre, sugars, protein), plus Image URL and Item Page Link. Nutrition is comprehensive — all values preserved.
Observed from CSV — recipes-ingredients-source
8 columns: Recipe_ID (int), Dish Name (redundant — already in recipes), Original Ingredient (messy free text like "0.17 g, Garam Masala Powder"), Category (clean: "Spices (Powder)", "Vegetables", "Oils/Fats" etc.), Quantity (numeric), Unit Original (inconsistent: gm/tsp(s)/tbsp), Standardised Ingredient (clean name), Unit Standardised (cleaner but still some variants). The Standardised Ingredient column is your ingredient library already — it just needs extracting.
Conversion decisions
1. Recipe_ID integer → "rcp_001" zero-padded string.
2. Extract all unique Standardised Ingredient values → ingredients.json with IDs "ing_001" etc.
3. Dish Name in ingredients CSV is redundant — drop it.
4. Original Ingredient column is noise — use Standardised Ingredient only.
5. Unit normalisation: gm→g, tsp(s)→tsp, tbsp(s)→tbsp.
6. Sub-Category in recipes is a cuisine sub-type ("Sabzi"), not a meal slot. Add a separate mealCategory column to your CSV (breakfast/lunch/dinner/snack) before converting.

Pocket Change

Pocket Change

Your budget CSV is a long flat table: one row per year/month/source/category/subcategory. The JSON pivots this into a nested structure — categories defined once, months as objects with budget and actual side by side. Far easier to query and chart.

Observed from CSV
6 columns: Years (int), Months (abbreviated: "Jan"), Source (Budget or Actual), Category, Sub-Category, Total (decimal). Each calendar month has multiple rows — one Budget row and one Actual row per sub-category. With 8 years × 12 months × ~6 categories × ~2–3 subcategories × 2 (budget + actual) = roughly 3,000–7,000 rows total. The structure is correct, just long.
Conversion decisions
1. Merge Years + Months into ISO year-month: "Jan" + 2017 → "2017-01".
2. Pivot Source from a row value into two sibling keys per subcategory: budget and actual. This halves row count and makes variance trivial to calculate.
3. Define categories and subcategories once in a master categories array — month spending references them by ID. Prevents typos drifting category names across 8 years.
4. Add tags array on each month for trip linkage — populated manually when a month contains significant travel spend.

Fuel Plan

Fuel Plan

Two very well-structured CSV files become two JSON files that work as a system. Phases define the targets; schedule is the day-by-day plan that references a phase. Your CSVs needed minimal restructuring — mainly grouping and adding cross-app link fields.

Observed — calorie-phases.csv
9 columns: label, sub (human date range string), start_date (ISO — already correct), end_date (ISO), avg_kcal, rest_kcal, run_kcal, long_kcal, description. Very clean. Phase 5 uses 2099-12-31 as an open-ended sentinel — make this explicit with a flag. The mode concept (base/marathon-build/taper/race/recovery/holiday) is implied but not present — add it as a new field.
Observed — calorie-schedule.csv
17 columns: date (ISO), label ("Mon 13"), activity, detail, type (walk/run), kcal, pb_g, alc (boolean), tagline, b_kcal, l_kcal, s_kcal, d_kcal, r_kcal (meal breakdown), tr_label, tr_detail (pre-run nutrition), zw_note (Zwift — empty in sample). Richly structured. The meal breakdown columns (b/l/s/d/r_kcal) group neatly into a mealTargets sub-object. Missing: actual logged data fields — these need adding for the app to function as a food log.
Conversion decisions
1. Add id to each phase: "phase_1" through "phase_5".
2. Phase 5 end_date: "2099-12-31" → endDate: null + openEnded: true.
3. Add mode field to each phase: base / marathon-build / taper / race / recovery / holiday. This drives "race mode" and "holiday mode" in the app later.
4. Add phaseId to each schedule day — derived from which phase's date range contains that date.
5. Group b/l/s/d/r_kcal columns → mealTargets{breakfast, lunch, snack, dinner, race}.
6. Rename pb_g → proteinBonusG. Rename kcal → targetKcal.
7. Add actual logging fields to each day: actualKcal, actualWeight, loggedMeals[] — all null until logged.
