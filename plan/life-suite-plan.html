Data Workflows

The hidden foundation. Good data habits make the apps genuinely useful and sustainable over years.

Recurring Workflows
1

Monthly Budget Update (App 03)

First weekend of each month. Takes ~5 minutes once the data entry form is built. Process:

  • Open the monthly update form in the Budget Tracker
  • Enter actuals by category from bank/card statements
  • Review any over-budget categories — add a note if it was exceptional
  • Save to the JSON dataset. Charts auto-update.
2

Daily Run Log (App 05)

Every run, within a few hours. Should take under 60 seconds. Process:

  • Open Training Planner → Daily Log
  • Today's plan is pre-shown — click to confirm or edit
  • Add: actual distance, time, notes, RPE (1–10)
  • Calorie burn auto-pushes to Calorie Planner
3

Daily Calorie Log (App 01)

Each evening or after each meal. The most frequent workflow — needs to be frictionless. Process:

  • Select meals from Recipe Book or add custom items
  • Running total vs target visible at all times
  • End-of-day: log weight if measuring
4

Trip Planning Setup (App 04)

As soon as a trip is confirmed — typically months ahead. Process:

  • Create trip entry with dates and destination
  • Add confirmed bookings with confirmation numbers
  • Clone relevant packing list template
  • Set budget estimate → links to Budget Tracker travel category
  • If marathon trip: link to race in Training Planner
5

Race Setup (App 05)

When you register for a marathon. Process:

  • Add race to calendar: date, location, distance, goal time
  • Build or load a training plan (12–20 weeks out)
  • Plan auto-populates week-by-week sessions counting back from race day
  • Create linked trip in Travel Planner if travelling for the race
6

Annual Review & Reset (All Apps)

Each January. Takes 1–2 hours but sets up the year well. Process:

  • Budget: Review prior year actuals, set this year's budget by category
  • Training: Review annual volume, enter target races for the year
  • Nutrition: Update calorie/macro targets if goals have changed
  • Travel: Enter all known trips; estimate costs; set travel budget
  • Recipes: Archive any recipes not used, add new ones
Data Architecture Decisions

Storage: localStorage vs JSON Files

ARCH

localStorage is quick to implement but browser-locked. JSON files saved to disk are portable and version-controllable with Git. Recommendation: JSON files with a simple save/load UI. This future-proofs the data.

Git for Version Control

ARCH

Store all app HTML files and JSON data in a single Git repo. Free backup, full history, easy to restore. A simple private GitHub repo is sufficient. Commit after each monthly update.

Single Source of Truth per Dataset

ARCH

Each app owns one JSON file as its source of truth. The homepage reads from each. No duplication. When a recipe changes in recipes.json, it's immediately reflected in the Calorie Planner.

Export / Backup Buttons

ARCH

Every app should have a Download JSON button that lets you save the current dataset. Build this from the start — one-click backup is better than hoping the browser hasn't cleared localStorage.

Guiding Principles

Rules of thumb to keep the suite coherent and sustainable as it grows over years.

🎯

Daily Usefulness First

Prioritise features you'll use every day (log a run, log food) over impressive but rare ones. Daily habits make an app indispensable.

⚡

Minimum Clicks to Log

Any data entry workflow should take under 60 seconds. If logging a run is slow or fiddly, you'll stop doing it. Design for speed.

📦

Data Portability Always

Every app exports its own data. You own your data. Never build yourself into a corner where years of runs or finances are locked in a single file with no backup.

🔗

Link, Don't Duplicate

A recipe lives in recipes.json once. The Calorie Planner reads it. The Meal Generator reads it. Change it in one place. Never copy data between apps manually.

🧱

MVP Before Enhancement

Build the simplest useful version of each app before adding features. A simple app that gets used beats a feature-rich one that's too complex to maintain.

🎨

One Style Guide to Rule Them

Add new UI patterns to the style guide as they appear. Never style an element ad-hoc in a single app. Consistency is what makes the suite feel like a product.

📅

Commit to a Maintenance Rhythm

Monthly budget update, weekly run log, annual review. Build the workflows around these rhythms so updating the apps feels like the right cadence, not a chore.

🌱

Build for Decades

Plain HTML, CSS and JSON will work in any browser in 20 years. Don't add dependencies you don't need. The budget data from 2025 should be just as accessible in 2045.
